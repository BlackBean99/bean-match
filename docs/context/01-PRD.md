# PRD - 소개팅 운영 플랫폼

## 1. 문서 개요
### 1.1 목적
본 문서는 소개팅 운영 플랫폼의 제품 요구사항을 정의한다.  
이 플랫폼은 소개 대상자, 주선자, 운영자가 소개팅 풀을 효율적으로 관리하고, 소개 진행 상태를 명확하게 파악하며, 사진과 프로필 데이터를 안정적으로 저장/운영할 수 있도록 지원한다.

### 1.2 제품 목표
1. 소개 모집인이 사용자 상태를 직관적으로 이해할 수 있어야 한다.
2. 한 사용자가 현재 소개 진행 중이면 다른 소개를 제안할 수 없어야 한다.
3. 주선자도 소개팅 풀 참가자가 될 수 있어야 한다.
4. 사진 업로드 및 관리가 서버 기준으로 안정적으로 동작해야 한다.
5. 소개 이력, 응답, 결과, 상태 변경이 추적 가능해야 한다.

### 1.3 핵심 사용자
- 소개 대상자(Participant)
- 주선자(Invitor)
- 운영자(Admin)

---

## 2. 문제 정의
기존에는 상태와 역할, 소개 진행 이력이 혼재되어 다음 문제가 발생했다.
- 사용자의 현재 소개 가능 여부를 빠르게 파악하기 어렵다.
- 주선자이면서 동시에 소개 대상자인 유저를 자연스럽게 표현하기 어렵다.
- 소개 건의 세부 진행 상태와 사용자 요약 상태가 섞여 운영자 해석이 달라진다.
- 사진 저장 방식이 설계되지 않으면 파일 유실, 경로 중복, 대표 사진 관리 문제가 생긴다.

## 3. 핵심 요구사항
### 3.0 2026-04-14 기획 기준
제품 방향은 “라운드 기반, 오픈 선택형, Operator 통제 매칭 시스템”으로 확장한다.

- `READY` 사용자만 라운드에 참여할 수 있다.
- `PROGRESSING` 사용자는 라운드 후보 노출과 신규 소개 생성에서 제외한다.
- `OpenLevel` 로 노출 범위를 제어한다.
  - `PRIVATE`: 운영자 큐레이션만
  - `SEMI_OPEN`: 제한된 풀 노출
  - `FULL_OPEN`: 전체 라운드 참여
- 한 라운드에서 사용자는 최대 2명까지만 선택할 수 있고, 선택은 직접 변경할 수 없다.
- 상호 선택은 자동 매칭 후보, 단방향 선택은 운영자 판단 대상으로 본다.
- 모집인(`INVITOR`)은 초대 링크로 신규 유입을 만들 수 있지만 매칭과 커뮤니케이션에는 개입하지 않는다.

### 3.1 사용자 상태는 모집인이 쉽게 이해할 수 있어야 한다
최종 사용자 상태는 다음과 같다.

- `INCOMPLETE` : 정보 미완성
- `READY` : 소개 가능
- `PROGRESSING` : 소개 진행 중
- `HOLD` : 잠시 보류
- `STOP_REQUESTED` : 탈퇴 요청
- `ARCHIVED` : 보관 완료
- `BLOCKED` : 운영 제한

### 3.2 소개 진행 중인 사용자는 다른 소개를 받을 수 없어야 한다
아래 중 하나의 소개 건이 활성 상태이면 사용자 상태는 `PROGRESSING` 이어야 한다.
- `OFFERED`
- `WAITING_RESPONSE`
- `MATCHED`
- `CONNECTED`
- `MEETING_DONE`
- `RESULT_PENDING`

즉, 사용자의 1:1 소개 결과가 확인되기 전까지 다른 소개 제안은 금지된다.

### 3.3 주선자는 역할(Role)로 관리한다
주선자는 상태가 아니라 역할이다.
- `PARTICIPANT`
- `INVITOR`
- `ADMIN`

한 사용자는 동시에 `PARTICIPANT` 와 `INVITOR` 를 가질 수 있다.

### 3.4 사진은 서버 파일시스템에 저장한다
- 원본 파일은 서버 파일시스템에 저장한다.
- DB에는 파일 메타데이터만 저장한다.
- 대표 사진과 일반 사진을 구분한다.
- 추후 썸네일 확장 가능 구조를 유지한다.

### 3.5 URL 공유 방식
- 사용자 온보딩: `/onboarding` 또는 모집인 초대 링크 `/invite/{invitorId}`
- 사용자 라운드 선택: `/rounds/{roundId}/participants/{userId}`
- 모집인 초대: `/invite/{invitorId}` 만 공유하며, 모집인은 매칭 관리 URL을 공유받지 않는다.
- 관리자 운영: `/users`, `/rounds`, `/matches`, `/users/{userId}` 를 내부 운영 URL로 사용한다.
- 연락처는 어떤 공유 URL에서도 `CONNECTED` 전까지 노출하지 않는다.

## 4. 기능 요구사항

### 4.1 사용자 프로필 관리
사용자는 다음 정보를 가진다.
- 이름
- 성별
- 상태
- 출생연도(또는 생년월일)
- 연락처
- 키
- 직업
- 회사명(선택)
- 본인소개
- 이상형(자유서술 + 구조화 조건)
- 프로필 사진 다수
- 대표 사진 1장
- 역할(Role)

### 4.2 이상형 관리
이상형은 다음 두 방식으로 관리한다.
1. 자유서술형
   - 이상형 설명
2. 구조화 조건
   - 선호 성별
   - 선호 나이 최소/최대
   - 선호 키 최소/최대
   - 선호 직업 텍스트
   - 선호 스타일/분위기 메모

### 4.3 소개 건 관리
소개 건은 다음을 관리해야 한다.
- 누가 누구를 소개하는지
- 누가 주선자인지
- 양쪽 응답 상태
- 현재 소개 건 상태
- 소개 결과
- 상태 변경 이력
- 운영 메모

### 4.4 사용자 상태 자동 반영
다음 로직이 필요하다.
- 사용자가 활성 소개 건을 가지면 `PROGRESSING`
- 결과 확정 후 재소개 의사가 있으면 `READY`
- 결과 확정 후 당분간 쉬고 싶으면 `HOLD`
- 탈퇴 요청 시 `STOP_REQUESTED`
- 3개월 후 `ARCHIVED`

### 4.5 관리자/운영자 화면 요구사항
운영자는 다음이 가능해야 한다.
- 전체 사용자 목록 조회
- 상태별 필터링
- 역할별 필터링
- 주선자별 소개 건 조회
- 소개 건 진행 상태 조회
- 응답 지연자 조회
- 대표 사진 변경
- 탈퇴 요청 대상 조회
- 보류 사용자 조회
- 상태 변경 이력 조회

## 5. 비기능 요구사항

### 5.1 성능
- 사용자 목록 필터링은 운영 규모 기준 즉시 검색 가능해야 한다.
- 대표 사진, 프로필 데이터 로딩은 운영 화면에서 과도하게 느리지 않아야 한다.
- 상태 계산은 조회마다 실시간 계산보다 저장 + 검증 혼합 전략을 권장한다.

### 5.2 보안
- 연락처는 소개 연결 시점 전까지 제한 노출 가능해야 한다.
- 사진 업로드는 MIME type 및 확장자 검증이 필요하다.
- 파일명은 서버 저장 시 UUID 기반으로 변경한다.
- 파일 다운로드는 인증된 사용자 또는 관리자만 가능하게 해야 한다.

### 5.3 감사 가능성
- 상태 변경 이력 저장
- 소개 건 이벤트 로그 저장
- 사진 업로드/삭제 이력 저장 가능 구조
- 누가 상태를 변경했는지 추적 가능해야 함

---

## 6. 성공 기준
- 모집인이 사용자 상태를 별도 설명 없이 이해할 수 있다.
- 소개 중복 제안이 발생하지 않는다.
- 주선자와 소개 대상자 역할이 충돌 없이 공존한다.
- 프로필/사진 데이터가 일관되게 저장된다.
- 운영자가 소개 건을 추적하고 결과를 확정할 수 있다.
