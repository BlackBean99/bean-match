# 운영 정책 / 관리자 기능 / 배치 설계

## 1. 운영 목적
운영자는 소개팅 풀을 안정적으로 유지하고, 중복 제안을 방지하며, 결과 확인을 통해 다음 상태를 정확히 반영해야 한다.

---

## 2. 운영자 핵심 화면

### 2.1 사용자 목록
필터:
- 상태
- 역할
- 성별
- 나이대
- 키 범위
- 직업 키워드
- 주선자 여부
- 프로필 완성 여부

표시 항목:
- 이름
- 성별
- 나이
- 키
- 직업
- 상태
- 역할
- 대표 사진 여부
- 최근 소개 진행 여부
- 최근 상태 변경 시각

운영 규칙:
- `PARTICIPANT` 회원은 기본 목록에 노출한다.
- `INVITOR` 계정도 운영 로그인 후 회원 목록/상세를 열람할 수 있다.
- `INVITOR` 는 오퍼 링크/토큰 공유까지 가능하지만, 사용자 수정/삭제/사진 편집/동기화는 할 수 없다.
- `PARTICIPANT` 와 `INVITOR` 역할을 동시에 가진 사용자는 둘 다 표시한다.

### 2.2 사용자 상세
탭 예시:
- 기본 정보
- 이상형
- 사진
- 소개 이력
- 상태 변경 이력
- 운영 메모

### 2.3 소개 건 목록
필터:
- 상태
- 주선자
- 결과
- 응답 지연 여부
- 생성일/종료일
- 검색어(참여자, 메모, 상태, 주선자)

표시 항목:
- 소개 건 ID
- 참여자 2명
- 주선자
- 현재 상태
- 최근 이벤트 시각
- 결과

### 2.4 결과 확인 화면
운영자는 만남 종료 후 다음을 입력한다.
- 결과: 성사 / 불발
- 추가 메모
- 재소개 의사
- 당분간 보류 여부
- 차단/주의 필요 여부

### 2.5 공유 URL 관리
운영자는 사용자 유형별로 다른 URL을 공유한다.

- 사용자 자동 노출 참여 진입: `/onboarding` 또는 `/invite/{inviteToken}`
- 사용자 개인 풀: `/pool/{userId}`
- 사용자 프로필 오퍼 열람: `/offer/pool/{userId}` + 관리자 발급 프로필 열람 토큰
- 참가자 개인 초대: `/invite/{inviteToken}`
- 관리자 운영: `/users`, `/rounds`, `/matches`, `/users/{userId}`

`/users/{userId}` 상세는 운영 로그인 후 접근한다. `INVITOR` 는 상세 열람과 오퍼 공유만 가능하고, `ADMIN` 만 편집/삭제/토큰 관리/동기화를 수행한다. 참가자 등록 시에는 개인 초대 토큰이 자동 발급되며, 사용자 상세에서는 카카오톡 안내 문구 복사 버튼으로 개인 초대 링크를 다시 생성할 수 있다.
관리자 삭제는 사용자 본문만 지우지 않고 관련 소개 건, 알림 subject 참조, 사진 자산까지 함께 정리해야 한다. 삭제 후 운영 화면은 `/users` 기준으로 다시 진입해 상태를 확인한다.
`/offer/pool/{userId}` 에서는 받은 관심을 확인하고, 선택을 수정하거나 전체 해제 상태로 다시 제출할 수 있어야 한다.
`/matches` 는 관심/소개 기록을 검색할 수 있어야 하고, 운영자는 특정 소개 기록을 삭제할 수 있어야 한다.

참가자와 모집인에게 관리자 운영 URL을 공유하지 않는다. 운영 화면은 운영 로그인 세션으로 보호한다. 참가자 라운드 선택 URL에는 연락처를 노출하지 않는다.
레거시 관리자 테스트 참여 URL은 실제 `round_selections` 데이터를 저장하지 않고 화면/후보 노출이 정상인지 확인하는 용도다.
프로필 열람 URL은 토큰 없이 직접 공유하지 않는다. 토큰은 사용자별로 발급/만료/해제한다.

## 3. 운영 정책

### 3.1 중복 소개 금지
한 사용자는 동시에 하나의 활성 소개만 가져야 한다.

### 3.2 결과 확인 전 재제안 금지
소개 결과 확인 전까지 다음 소개를 생성할 수 없다.

### 3.3 상태 가시성 우선
모집인이 빠르게 판단할 수 있도록 `PROGRESSING` 상태를 적극 사용한다.

### 3.3.1 자동 노출 정책
- `READY` + (`SEMI_OPEN` 또는 `FULL_OPEN`) + 프로필 노출 동의 사용자만 자동 노출 후보로 노출한다.
- `PRIVATE` 사용자는 운영자 큐레이션 대상으로만 관리한다.
- 신규 가입자는 자동 노출 풀에서 최대 3명의 관심을 제출한다.
- 기존 회원은 새 멤버당 최대 1회 관심을 제출한다.
- `PROGRESSING` 사용자는 어떤 자동 노출 후보에서도 제외한다.
- 활성 incoming 관심이 5건 이상인 사용자는 자동 노출 우선순위에서 뒤로 밀거나 일시 제외한다.
- 운영자는 사용자별로 자동 노출 일시중지, 오픈 레벨, 알림 수신 여부를 조정할 수 있다.
- 프로필 열람 후보는 같은 성별을 제외하고, `READY` + 노출 동의 상태 사용자만 보여준다.
- 프로필 열람 토큰은 소개 생성과 연락처 확인 권한은 주지 않지만, 최대 3명까지 관심 제출은 허용한다.
- 제출된 관심은 재제출로 수정 가능하며, 아무 것도 선택하지 않은 상태로도 저장해 기존 관심을 철회할 수 있다.

### 3.4 탈퇴 요청 처리
- 즉시 `STOP_REQUESTED`
- 실운영 소개 제외
- 90일 후 `ARCHIVED`

### 3.5 차단 정책
예시 사유:
- 허위 프로필
- 노쇼 반복
- 운영 방침 위반
- 민원 다수
- 부적절한 사진/소개글

## 4. 배치/자동화 설계

### 4.1 탈퇴 요청 보관 배치
주기: 하루 1회

로직:
1. `STOP_REQUESTED` 상태 사용자 조회
2. `stop_requested_at + 90일 <= now`
3. `ARCHIVED` 전환
4. 필요 시 연락처/사진 접근 제한 강화

### 4.2 소개 응답 만료 배치
주기: 1시간 또는 하루 1회

로직:
1. 응답 기한이 지난 `WAITING_RESPONSE` 조회
2. `EXPIRED` 처리
3. 참여자의 사용자 상태 재평가
4. 이벤트 로그 저장

### 4.3 사용자 상태 정합성 배치
주기: 하루 1회

로직:
1. 활성 소개 건이 있는 사용자 조회
2. `status != PROGRESSING` 인 경우 교정
3. 활성 소개 건이 없는 `PROGRESSING` 사용자 조회
4. 운영 규칙에 따라 `READY/HOLD` 후보로 알림 또는 교정

### 4.4 관심 만료 배치
주기: 하루 1회

로직:
1. `Interest.status = ACTIVE` 인 레코드 조회
2. `expires_at < now` 또는 생성 후 21일 경과 레코드 만료 처리
3. 상호 관심 후보가 이미 소개로 전환된 경우 `CONVERTED_TO_INTRO` 상태 유지
4. 관리자 대시보드 지표 재계산

## 6. 운영상 주의사항

### 6.1 상태와 역할을 혼동하지 말 것
- 상태: 지금 소개 가능한가?
- 역할: 이 사람이 어떤 일을 할 수 있는가?

### 6.2 대표 사진 관리
대표 사진이 없거나 여러 장으로 꼬이지 않도록 트랜잭션 처리 필요
대표 사진을 지정하면 해당 사진이 1번으로 재정렬되도록 유지한다.

### 6.3 연락처 공개 시점
연락처는 `CONNECTED` 전에는 공개 금지 정책을 권장

### 6.4 사진 서버 운영
- 백업 정책 필요
- 중복 파일명 방지
- 파일 정리 정책 필요
- soft delete 이후 미사용 파일 정리 배치 고려

## 7. MVP 범위와 추후 확장

### MVP
- 사용자 등록/수정
- 역할 관리
- 이상형 관리
- 사진 업로드/대표사진 설정
- 소개 생성/응답/결과 확인
- 사용자 상태 관리
- 운영자 조회 화면
- 관리자 프로필 열람 토큰 발급/해제와 후보 열람 게이트

- 썸네일 생성
- 이미지 검수
- 연락처 마스킹/부분 노출
- 자동 매칭 추천
- 운영 알림 시스템
- 감사 로그 대시보드
