# 운영 정책 / 관리자 기능 / 배치 설계

## 1. 운영 목적
운영자는 소개팅 풀을 안정적으로 유지하고, 중복 제안을 방지하며, 결과 확인을 통해 다음 상태를 정확히 반영해야 한다.

---

## 2. 운영자 핵심 화면

### 2.1 사용자 목록
필터:
- 상태
- 역할
- 성별
- 나이대
- 키 범위
- 직업 키워드
- 주선자 여부
- 프로필 완성 여부

표시 항목:
- 이름
- 성별
- 나이
- 키
- 직업
- 상태
- 역할
- 대표 사진 여부
- 최근 소개 진행 여부
- 최근 상태 변경 시각

### 2.2 사용자 상세
탭 예시:
- 기본 정보
- 이상형
- 사진
- 소개 이력
- 상태 변경 이력
- 운영 메모

### 2.3 소개 건 목록
필터:
- 상태
- 주선자
- 결과
- 응답 지연 여부
- 생성일/종료일

표시 항목:
- 소개 건 ID
- 참여자 2명
- 주선자
- 현재 상태
- 최근 이벤트 시각
- 결과

### 2.4 결과 확인 화면
운영자는 만남 종료 후 다음을 입력한다.
- 결과: 성사 / 불발
- 추가 메모
- 재소개 의사
- 당분간 보류 여부
- 차단/주의 필요 여부

### 2.5 공유 URL 관리
운영자는 사용자 유형별로 다른 URL을 공유한다.

- 사용자 온보딩: `/onboarding` 또는 `/invite/{invitorId}`
- 사용자 라운드 선택: `/rounds/{roundId}/participants/{userId}`
- 모집인 초대: `/invite/{invitorId}`
- 관리자 운영: `/users`, `/rounds`, `/matches`, `/users/{userId}`

참가자와 모집인에게 관리자 운영 URL을 공유하지 않는다. 참가자 라운드 선택 URL에는 연락처를 노출하지 않는다.

## 3. 운영 정책

### 3.1 중복 소개 금지
한 사용자는 동시에 하나의 활성 소개만 가져야 한다.

### 3.2 결과 확인 전 재제안 금지
소개 결과 확인 전까지 다음 소개를 생성할 수 없다.

### 3.3 상태 가시성 우선
모집인이 빠르게 판단할 수 있도록 `PROGRESSING` 상태를 적극 사용한다.

### 3.3.1 라운드 노출 정책
- `READY` + `FULL_OPEN` 사용자만 전체 라운드 후보로 노출한다.
- `PRIVATE` 사용자는 운영자 큐레이션 대상으로만 관리한다.
- `SEMI_OPEN` 사용자는 제한된 풀 또는 운영자 판단 대상으로 관리한다.
- `PROGRESSING` 사용자는 어떤 라운드 후보에서도 제외한다.
- 과다 선택 사용자는 운영자가 다음 라운드 노출을 제한할 수 있다.

### 3.4 탈퇴 요청 처리
- 즉시 `STOP_REQUESTED`
- 실운영 소개 제외
- 90일 후 `ARCHIVED`

### 3.5 차단 정책
예시 사유:
- 허위 프로필
- 노쇼 반복
- 운영 방침 위반
- 민원 다수
- 부적절한 사진/소개글

## 4. 배치/자동화 설계

### 4.1 탈퇴 요청 보관 배치
주기: 하루 1회

로직:
1. `STOP_REQUESTED` 상태 사용자 조회
2. `stop_requested_at + 90일 <= now`
3. `ARCHIVED` 전환
4. 필요 시 연락처/사진 접근 제한 강화

### 4.2 소개 응답 만료 배치
주기: 1시간 또는 하루 1회

로직:
1. 응답 기한이 지난 `WAITING_RESPONSE` 조회
2. `EXPIRED` 처리
3. 참여자의 사용자 상태 재평가
4. 이벤트 로그 저장

### 4.3 사용자 상태 정합성 배치
주기: 하루 1회

로직:
1. 활성 소개 건이 있는 사용자 조회
2. `status != PROGRESSING` 인 경우 교정
3. 활성 소개 건이 없는 `PROGRESSING` 사용자 조회
4. 운영 규칙에 따라 `READY/HOLD` 후보로 알림 또는 교정

## 5. 운영 KPI 예시
- 상태별 사용자 수
- 월간 소개 생성 수
- 주선자별 소개 성사율
- 응답률
- 평균 응답 시간
- 만남 완료 후 결과 회수율
- `PROGRESSING` 장기 체류 사용자 수
- 탈퇴 요청 후 보관 전환율

## 6. 운영상 주의사항

### 6.1 상태와 역할을 혼동하지 말 것
- 상태: 지금 소개 가능한가?
- 역할: 이 사람이 어떤 일을 할 수 있는가?

### 6.2 대표 사진 관리
대표 사진이 없거나 여러 장으로 꼬이지 않도록 트랜잭션 처리 필요

### 6.3 연락처 공개 시점
연락처는 `CONNECTED` 전에는 공개 금지 정책을 권장

### 6.4 사진 서버 운영
- 백업 정책 필요
- 중복 파일명 방지
- 파일 정리 정책 필요
- soft delete 이후 미사용 파일 정리 배치 고려

## 7. MVP 범위와 추후 확장

### MVP
- 사용자 등록/수정
- 역할 관리
- 이상형 관리
- 사진 업로드/대표사진 설정
- 소개 생성/응답/결과 확인
- 사용자 상태 관리
- 운영자 조회 화면

### 추후 확장
- 썸네일 생성
- 이미지 검수
- 연락처 마스킹/부분 노출
- 자동 매칭 추천
- 운영 알림 시스템
- 감사 로그 대시보드
