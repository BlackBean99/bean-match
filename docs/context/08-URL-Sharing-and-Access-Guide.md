# URL Sharing and Access Guide

## 목적
2026-05-05 자동 노출 요구사항 기준으로 사용자, 모집인, 관리자가 서로 다른 URL을 받도록 운영한다. URL은 역할별 목적을 분리하고, 연락처와 운영자 기능을 불필요하게 노출하지 않기 위한 경계다.

## 기본 전제(호스트)
- 로컬 개발: `http://localhost:3000`
- 같은 LAN 테스트: `npm run dev` 출력의 `Network` 주소 사용
- 외부 퍼블리싱(도메인/공개 IP)은 별도 배포 구성이 필요하다. (`docs/context/09-Deployment-and-Env.md`)

## 사용자(PARTICIPANT)
사용자는 자동 노출 참여용 토큰 링크와 개인 풀 URL만 받는다.

- 자동 노출 참여 진입: `/onboarding/access/{token}`
- 운영자 토큰 발급 화면: `/users/{userId}` 상세의 온보딩 접근 토큰 관리자
- 개인 풀: `/pool/{userId}`
- 프로필 오퍼 토큰 페이지: `/offer/pool/{userId}`

자동 노출 참여 진입 URL에서 가능한 일:
- 운영자가 발급한 토큰으로 참가자 식별
- 자동 노출 레벨 선택 (`PRIVATE`, `SEMI_OPEN`, `FULL_OPEN`)
- 프로필 노출 동의와 신규 멤버 알림 수신 여부 결정
- 기존 사용자 데이터의 상태와 자동 노출 큐 갱신
- 개인 풀 URL로 이동

사용자 개인 풀 URL에서 가능한 일:
- 신규 가입자일 때 자동 노출 후보를 보고 최대 3명까지 관심 표시
- 기존 회원일 때 새 멤버 알림 카드 확인
- 알림 카드에서 새 멤버에게 최대 1번 관심 표시
- 운영자 승인 전까지 연락처 없이 프로필만 확인

사용자 개인 풀 URL에서 금지되는 일:
- 연락처 확인
- 관리자 메모 확인
- 다른 사용자의 관심 데이터 확인
- 소개 승인/거절

토큰 기반 프로필 열람 URL에서 가능한 일:
- 관리자 발급 프로필 열람 토큰 입력 후 후보 열람
- 본인과 같은 성별을 제외한 소개 가능 후보 프로필 확인
- 최대 3명까지 관심 제출
- 연락처 없이 사진/자기소개/기본 프로필만 확인

토큰 기반 프로필 열람 URL에서 금지되는 일:
- 소개 생성
- 연락처 확인
- 관리자 메모 확인

## 모집인(INVITOR)
모집인은 신규 유저 유입을 담당하며, 운영 접근 코드가 있으면 관리자와 동일한 운영 화면을 사용한다.

- 초대 출처 표기: 토큰 발급 시 운영 메모로만 기록
- 직접 입장 URL은 제공하지 않는다.
- 모집인도 운영 접근 코드로 `/users`, `/matches`, `/rounds`, `/users/{userId}` 를 동일하게 본다.

모집인이 할 수 있는 일:
- 신규 사용자에게 운영자가 발급한 토큰 링크를 전달하도록 요청

모집인이 할 수 없는 일:
- 매칭 조율
- 후보 재배치
- 라운드 선택 데이터 확인
- 사용자 연락처 확인
- 커뮤니케이션 개입

## 관리자(ADMIN / Operator)
관리자는 내부 운영 URL을 사용한다.

- 사용자 풀: `/users`
- 사용자 상세: `/users/{userId}`
- 자동 노출 운영: `/rounds`
- 매칭 조율: `/matches`
- Notion 동기화: 운영 화면의 `동기화` 버튼
- 사용자 상세에서 프로필 열람 토큰 발급/해제

관리자가 할 수 있는 일:
- 자동 노출 큐 확인
- 관심 데이터 확인
- 상호 관심 소개 후보 승인/반려/소개 전환
- 단방향 관심의 운영자 판단
- 노출 제한과 후보 재배치
- `CONNECTED` 이후 연락처 공개 정책 처리
- 필요 시 기존 라운드 테스트 URL로 레거시 화면 점검

## 공유 원칙
- 공개 공유 URL은 `/onboarding/access/{token}`, `/offer/pool/{userId}` 이다.
- `/onboarding` 과 `/invite/{invitorId}` 는 직접 입장용이 아니라 토큰 링크 안내용 화면이다.
- `/invite/{invitorId}` 는 모집인 출처를 확인하는 안내 화면이다.
- 관리자 URL은 접근 코드가 있는 내부 운영자에게만 공유한다.
- 연락처는 `CONNECTED` 전까지 어떤 URL에서도 노출하지 않는다.
- `PROGRESSING` 사용자는 자동 노출 후보와 신규 소개 생성에서 제외한다.
- `/offer/pool/{userId}` 는 관리자 발급 프로필 열람 토큰을 통과해야만 진입할 수 있다.
- 개인 입장 공유는 `userId` 쿼리 문자열이 아니라 발급 토큰으로만 전달한다.

## 외부 사용자 신규 이성 리스트 확인 가이드
### 1. 신규 가입자가 직접 후보를 보는 흐름
1. 운영자는 사용자 상세 화면에서 온보딩 접근 토큰을 발급한다.
2. 운영자는 `/onboarding/access/{token}` 링크를 참가자에게 개인적으로 전달한다.
3. 참가자는 토큰 링크에서 이름을 확인하고 필요 시 수정해 본인 확인을 마친다.
4. 참가자는 자동 노출 레벨을 선택한다.
5. `SEMI_OPEN` 또는 `FULL_OPEN` 을 선택했다면 프로필 노출 동의를 켠다.
6. 저장이 끝나면 시스템은 `/pool/{userId}` 로 이동시킨다.
7. 참가자는 개인 풀에서 본인과 반대 성별의 소개 가능 후보를 보고 최대 3명까지 관심을 남긴다.

신규 가입자에게 보여야 하는 화면:
- 후보 카드의 사진, 나이/키/직업/회사/소개/이상형
- 최대 3명 선택 안내
- 연락처 비노출 안내

신규 가입자에게 보이면 안 되는 정보:
- 전화번호와 외부 연락 수단
- 관리자 메모
- 다른 사용자의 관심 내역
- 소개 승인/반려 같은 운영 액션

### 2. 기존 회원이 새 이성 멤버를 보는 흐름
1. 기존 회원도 자신의 `/pool/{userId}` 를 연다.
2. 새로 풀에 들어온 반대 성별 멤버가 있으면 상단 알림 카드가 뜬다.
3. 기존 회원은 알림 카드마다 최대 1회 관심을 남길 수 있다.
4. 상호 관심이 생겨도 운영자 승인 전까지는 연락처가 열리지 않는다.

### 3. 외부 공유용 프로필 열람 흐름
1. 운영자는 사용자 상세에서 프로필 열람 토큰을 발급한다.
2. 외부 유저는 `/offer/pool/{userId}` 에 진입한다.
3. 발급받은 토큰을 입력하면 본인과 같은 성별을 제외한 후보 목록을 본다.
4. 관심 있는 상대를 최대 3명까지 선택해 제출한다.

프로필 열람에서 가능한 일:
- 후보 프로필 보기
- 사진 보기
- 최대 3명까지 관심 제출
- 기본 정보와 자기소개 읽기

프로필 열람에서 불가능한 일:
- 소개 생성
- 연락처 확인
- 운영 메모 확인

## 운영자 점검 체크리스트
외부 사용자가 신규 이성 리스트를 못 본다면 아래 순서로 확인한다.

1. 해당 사용자의 상태가 `READY` 인지 확인한다.
2. 해당 사용자의 `openLevel` 이 `FULL_OPEN` 또는 정책상 허용된 `SEMI_OPEN` 인지 확인한다.
3. `PARTICIPANT` 역할이 붙어 있는지 확인한다.
4. 대표 사진이 있거나 최소한 프로필 카드에 필요한 기본 정보가 채워져 있는지 확인한다.
5. 대상 후보들이 `PROGRESSING` 이 아니고, 기존 활성 소개와 충돌하지 않는지 확인한다.
6. 같은 성별만 남아 있지 않은지 확인한다.
7. 개인 풀 URL이 맞는지, 즉 `/pool/{userId}` 의 `userId` 가 실제 사용자 ID와 일치하는지 확인한다.
8. 프로필 열람 공유라면 `/offer/pool/{userId}` 와 발급 토큰이 같이 전달됐는지 확인한다.

## Supabase 복구 메모
프로젝트를 resume 한 뒤 동기화는 성공했는데 `/users`, `/pool/{userId}`, 온보딩 저장에서 아래 오류가 나면 스키마가 덜 올라온 상태다.

```text
Supabase REST 400: {"code":"42703","message":"column users.exposure_consent does not exist"}
```

원인:
- `supabase/migrations/20260505000000_auto_exposure_interest_flow.sql` 이 현재 프로젝트에 아직 반영되지 않았다.

즉시 조치:
1. Supabase 프로젝트를 resume 한다.
2. 프로젝트를 다시 link 한다.
3. `supabase db push` 또는 SQL Editor 로 `20260505000000_auto_exposure_interest_flow.sql` 을 반영한다.
4. 필요하면 `npm run sync:notion -- --write` 를 다시 실행한다.

현재 앱의 임시 호환 동작:
- 누락 컬럼이 있어도 회원 목록과 후보 열람은 최대한 표시한다.
- `exposure_consent` 가 없으면 `openLevel !== PRIVATE` 를 동의로 간주한다.
- `new_member_notifications_enabled` 가 없으면 노출 동의 상태와 동일하게 간주한다.
- `exposure_paused` 가 없으면 일시중지는 꺼진 상태로 간주한다.

다만 위 호환은 화면 복구용이다. 운영 기준값을 정확히 저장하려면 마이그레이션 반영이 필요하다.
